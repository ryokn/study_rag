# プロジェクトルール

## ブランチ戦略

- 機能を実装する際は必ず `feature/<機能名>` ブランチを作成してから実装すること
- `main` ブランチへの直接コミットは禁止

## 動作確認

- 実装後は必ず動作確認を実施すること
- CLI機能はコマンド実行で動作を確認すること
- Web UI機能はブラウザで画面を確認すること

## コーディングスタイル

### 基本原則（common/coding-style.md 準拠）

- **KISS**: 最もシンプルな解を選ぶ。可読性を最優先にする
- **DRY**: 繰り返しのロジックは関数に切り出す
- **YAGNI**: 今必要でない機能・抽象化は作らない
- **イミュータビリティ**: オブジェクトを直接変更しない。`{**state, "key": value}` のように新しい値を返す
- **早期リターン**: ネストが深くなる前に条件分岐で早めに返す

### Python規約（python/coding-style.md 準拠）

- **PEP 8** に従う（命名・インデント・空行）
- **型アノテーション**をすべての関数シグネチャに付与する
- 定数は `UPPER_SNAKE_CASE` で定義する

### コメント方針（学習目的プロジェクト）

このプロジェクトは学習目的であるため、**通常より多めのコメントを付与すること**。

- **モジュール冒頭**: そのファイルの役割・使用ライブラリの概念・処理フローを説明する
- **関数**: Args / Returns を記述し、「なぜその実装か」の背景も説明する
- **非自明な処理**: ライブラリの制約・設計上の理由・ハマりポイントをインラインコメントで補足する

### 関数サイズの目安

- 1関数は50行以内を目安にする
- 1ファイルは800行を超えないようにする
